ARCHITECTURE FIXES (2026-04-25) — Production Stability
--------------------------------------------------------------

[FIX 1] SNS Webhook Signature Verification
  File: platform/api/routes/webhooks.py
  What was broken:
    Anyone who knew the /webhooks/ses URL could POST fake bounce events and
    mass-unsubscribe all tenants' contacts with no authentication.
  What was fixed:
    Every request to /webhooks/ses now verifies the AWS SNS RSA signature
    before any data is read or mutated. Fake requests → HTTP 403 instantly.
  Behaviour by environment:
    ENVIRONMENT=development  → verification SKIPPED (curl testing still works)
    ENVIRONMENT=production   → verification REQUIRED (invalid sig → 403)
  Dependency: 'cryptography' package (already installed via python-jose[cryptography])
  SubscriptionConfirmation: auto-confirmed by HTTP GET to SubscribeURL (no manual step needed)

[FIX 2] CORS Already Locked (verified 2026-04-25)
  File: platform/api/main.py (line 156)
  Status: ALREADY CORRECT — CORS reads from FRONTEND_URL env var, not wildcard.
  Current value in .env: FRONTEND_URL=http://localhost:3000
  For production: set FRONTEND_URL=https://your-app.com in your deployment env.

[FIX 3] Bounce Rate Redis Writer (Circuit Breaker Now Has Real Data)
  File: platform/api/routes/webhooks.py — _write_bounce_to_redis()
  What was broken:
    The circuit breaker logic checked Redis key `tenant:{id}:bounces:rolling`
    but this key was NEVER written anywhere. The circuit breaker was completely
    deaf — your AWS SES account could be suspended before you noticed a spike.
  What was fixed:
    After every confirmed hard bounce from SES, the key is incremented:
      INCR tenant:{tenant_id}:bounces:rolling
      EXPIRE tenant:{tenant_id}:bounces:rolling 2592000  (30-day rolling window)
    The circuit breaker now has actual data to act on.
  Note: requires tenant_id to be set as a message tag in your SES configuration.

[FIX 4] Standalone Scheduler with Redis Distributed Lock
  File: platform/worker/scheduler.py
  What was broken:
    The campaign scheduler ran as a background asyncio task INSIDE FastAPI.
    Running 2 API replicas → 2 scheduler instances → every scheduled campaign
    fires TWICE. No guard existed to prevent this.
  What was fixed:
    - scheduler.py is now a proper standalone process with a Redis distributed lock:
        SET scheduler:global:lock {instance_uuid} NX EX 90
      Only ONE instance across all processes/replicas can hold this lock.
    - Lock release uses a Lua script to avoid releasing a lock owned by another instance.
    - Scheduler writes a heartbeat key every cycle:
        SET scheduler:heartbeat {timestamp} EX 120
      If this key expires → scheduler is dead → monitoring can alert.
    - The embedded scheduler in FastAPI now defaults to DISABLED:
        ENABLE_EMBEDDED_CAMPAIGN_SCHEDULER=false  (was 'true')
      A ⚠️ WARNING log fires if you accidentally enable both.
