Email Worker Architecture (PostgreSQL as a Queue)

This document explains the architecture for processing email campaigns in the background using Python background scripts (Workers) and PostgreSQL. This avoids the need for complex external tools like Redis or Kafka at this startup scale.

Simple Explanation in Telugu (Tanglish)

Mana system lo thousands of emails elaa velthayo step-by-step ikkada vundi:

Step 1: User Input (Send Button Clicked)

  • User vachi "Send to 10,000 people" ani click chestharu.
  • Frontend (Next.js) nunchi Backend (FastAPI) ki request velthundi.
  • Backend ventane pani kosam user ni aagipomani cheppakunda, PostgreSQL database loni email_tasks table lo 10,000 rows ni "PENDING" status to create chestundi.
  • Create avagane Frontend ki "Done" ani cheppesthundi. User ki ventane Progress bar (0/10000) kanipistundi.

Step 2: Background Workers Holding the Load (The Magic)

Asalu email pampedi ee background scripts (workers) ye:

  • Manam server lo 3, ledha anthakante ekkuva "Worker Scripts" (Background Python processes) ni on chestham.
  • Anni workers okey sari database meeda padithe oka user ki 3 sarlu same email vellipothundhi kadha? Ala avvakunda PostgreSQL lo SKIP LOCKED ane special command vadutham.
  • Worker 1 velli DB ni aduguthundi: "Naku first 50 PENDING emails ivvu, inkenduku padithe valu lock cheyyaku".
  • DB 1st 50 isthundi. Ventane Worker 2 velli inko 50 emails techkuntundi, avi old vatitho match avvavu.
  • Workers aakali meeda padi aa 50 emails ni fast ga Amazon SES (Email pampise network) ki pamputhayi.
  • SES nunchi response ochaka (Sent ayyinda, Bounce ayinda), DB loki ochi aa rows ni "PENDING" nunchi "SENT" leda "FAILED" ga marchesuthayi.
  • Idi every 2 seconds ki automatic ga loop lo jarugutu ne untundi, till aa 10,000 emails aypoyentha varaku.

Step 3: User Dashboard (Watching Progress)

  • Worker aa lopala aa 10,000 emails processing chestunappudu, mana website (Next.js) every 3-5 seconds ki backend ni "Yenti progress entha aypoindi" ani background lo aduguthundi.
  • Adi DB lo lekkesi 3000 emails "SENT" ayyayi ani number isthe, User ki beautiful progress bar 30% daggara move avvuthundi.

Deeni valla upayogalu?

  • Crash Proof: System eppudu crash avvadu (Entha mandiki push kottina, load antha queue lone pothundi, server hand ivvadu).
  • Cost Effective: Kotha kotha tools like Redis / Celery maintain chese pani lekunda, already unna database ne queue ga clever ga vadeskumuntunnam!
